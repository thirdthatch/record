19:35
詳しく説明します。現状の実装には3つの根本的な問題があります。

問題1：サンプル密度が致命的に不足
現状
ptsPerRev = 180点 / 回転
1回転 = 180点 → 1点あたり 2° の角度間隔

必要な値
レコードの針は 20kHz まで再生します。 33RPMで20kHzを記録するには：

1秒間の回転数 = 33/60 = 0.55 回転/秒
20kHz = 20,000サイクル/秒
→ 1回転あたり必要なサンプル数 = 20,000 / 0.55 ≈ 36,000点/回転
現状の180点は必要な200分の1以下です。これでは1kHz以上の音はほぼ消えます。

問題2：ステレオ溝がIEC規格に非準拠
現状の実装（間違い）
javascript
zm = sl * depth * 0.4          // 左=垂直変調
r += sr_ * g.w * 0.15          // 右=ラテラル変調（半径方向）
正しいIEC 60098規格（45/45方式）
左チャンネル = 内壁を45°方向に変調
右チャンネル = 外壁を45°方向に変調
Left  壁の変位 = (L + R) / √2  ← ラテラル成分
Right 壁の変位 = (L - R) / √2  ← 垂直成分
溝断面を見ると：

    外側
   /    \
  / 45° 45°\
 /  右  左  \
/____________\
    内側
現在の実装は左右を「垂直」と「半径方向」に分けており、針が物理的に読み取れない方向に変調しています。

問題3：RIAA等化が未実装
実際のレコード製造プロセス
原音 → [RIAAイコライザー] → カッティング → レコード盤
再生時: レコード盤 → [逆RIAA] → アンプ → スピーカー
RIAAカーブは：

高音域（2kHz以上）を大幅にブーストしてカット
低音域（500Hz以下）を大幅にカットしてカット
理由：

低音をそのままカットすると溝の振幅が数mmになり物理的に入らない
高音をブーストして記録することで再生時のS/N比を改善
これなしでは低音が過大振幅でディスクに収まらず、高音はノイズに埋もれます。

解決策のまとめ
問題	現状	必要	難易度
サンプル密度	180点/回転	~36,000点/回転	★★（STLサイズが巨大化）
ステレオ規格	独自方式	IEC 45/45方式	★★★
RIAA等化	なし	必須	★★（DSP実装）
STLファイルサイズ	数MB	数百MB〜数GB	★★★（要分割出力）
⚠️ 最大の課題：STLファイルサイズ
高密度にすると：

12インチ, 33RPM
→ 約500回転 × 36,000点 = 1,800万点
→ 三角形数 ≈ 7,200万個
→ STLファイルサイズ ≈ 3.6GB
ブラウザでのリアルタイム生成は不可能になります。
