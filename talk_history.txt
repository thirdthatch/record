## 現在の流れ

### 1. STLファイルから音声ファイル生成の流れ
- record_gui.html の UI から `.stl` ファイルを選択 / ドロップすると、`uploadStlFile()` が呼ばれます。
- `uploadStlFile()` は `/api/reverse_record` に `multipart/form-data` で POST 送信します。
- gui_server.py の `GUIHandler._handle_reverse_record()` で受け取ります。
- サーバー側では stl_to_audio.py の以下の処理を使います。
  - `load_stl_vertices()` で STL 頂点を読み込み
  - `side_vertices_by_top_z()` でレコード面を抽出
  - `infer_size_rpm_quality()` でサイズ・RPM・品質・ピッチを推定
  - `reconstruct_waveform()` で溝データから左右波形を再構成
  - `write_wav()` で WAV ファイルを書き出し
- 結果として `wav_url` と `waveform` データが返り、UI 側で再生・波形表示します。

### 2. 生成したレコード STL の自動検証
- `STL出力` 生成後、出力ファイルリストに `.stl` が含まれていれば、`showOutput()` で自動的に `fetchStlAndVerify()` を呼び出します。
- これにより、生成済みの `.stl` をダウンロードしてそのまま再度 `uploadStlFile()` で検証・WAV化します。

## 探索結果

### 1. クリック再生位置の問題
- record_gui.html の `seekWaveform()` は本来 `audioPlayer.currentTime` をクリック位置に設定しています。
- ただし、クリック位置の算出で `rect.width` に対するクリッピングが不足していたため、まれに不正な `ratio` になる可能性がありました。
- この部分を修正して、`clientX` 位置をキャンバス幅内にクランプし、`ratio` が数値でない場合は無視するようにしました。

### 2. 逆レコード読み取りの本質的な問題
- stl_to_audio.py の `sort_spiral_points()` で、STL 内の螺旋点を並べる際に生の `atan2` 角度をそのまま使っていました。
- 生成側のレコードは時計回りの螺旋になっているため、角度を `0..2π` に正規化してからソートする必要があります。
- ここがズレていると、時間軸順に並ぶべきデータが乱れ、再生結果が「右から左に移動するザー音」のような不正な音になる可能性が高いです。

### 3. 出力ファイルはモックではありません
- gui_server.py 経由で stl_to_audio.py が実際に STL を読み取り、`write_wav()` で WAV を生成しています。
- つまり、出力ファイルは「モック」ではなく、実際の復元処理結果です。
- その結果がノイズっぽい場合は、STL の読み取り／復号アルゴリズム側に問題がある可能性が高いです。

## 根本原因の探索結果

### 1. ダウンロード画面が出ない本質
record_gui.html のダウンロードパネルは、`showOutput(result)` が呼ばれて初めて表示されます。  
そのため、`/api/generate` が `400` で失敗すると、単純に詳細画面は表示されません。

> つまり、UI の表示不具合ではなく、先に API が失敗している点が根本原因です。

---

### 2. 生成リクエストが `400` になる原因候補
gui_server.py の `_handle_generate()` は、次の条件で `400` を返します。

- `Content-Type` が `multipart/form-data` ではない
- `audio` フィールドが存在しない
- `size`, `rpm`, `groove`, `sides`, `split_mode`, `multi_files_mode` に不正な値がある
- `groove_spacing_factor` が `0.5〜2.0` の範囲外

このうち最も怪しいのは、`cgi.FieldStorage` の初期化です。
元のコードでは `CONTENT_LENGTH` を `environ` に渡しておらず、multipart 解析が不安定になる可能性があります。

- gui_server.py 155 行付近
- `cgi.FieldStorage(..., environ={...})`

---

### 3. ダウンロード時の重大な根本原因
gui_server.py の `_send_file()` は、ファイルを丸ごとメモリに読み込んで送信しています。

```python
data = path.read_bytes()
self.wfile.write(data)
```

これには重大な問題があります。

- 大きな `.wav` / `.stl` / `.3mf` をダウンロードするとメモリを大量消費
- 実際に `Exit code 137` でサーバープロセスが強制終了している
- さらに `BrokenPipeError` が大量に出ている

この部分がダウンロード失敗とプロセス殺害の直接的な原因です。

---

### 4. クライアント切断時の例外処理不足
`_send_output()` や `_send_json()` でクライアントが切断した場合の例外を十分に捕捉していないため、ログに大量の `BrokenPipeError` が出ています。
